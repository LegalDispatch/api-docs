# LegalDispatch API Documentation

> Unified API documentation for the LegalDispatch platform — all public-facing endpoints in one place.

## Overview

This repo holds the **unified OpenAPI specification** and **narrative guides** for the LegalDispatch API. Individual service specs flow in via CI/CD and are merged into a single spec that powers the interactive docs at `api.legaldispatch.dev/docs`.

### Two-Layer Documentation Model

| Layer | Audience | Contents |
|-------|----------|----------|
| **Unified Gateway Docs** (this repo) | Frontend devs, QA, external consumers | Public endpoints only (`/api/v1/*`, `/admin/v1/*`) |
| **Individual Service Docs** (service repos) | Backend devs, DevOps | All routes, internal APIs, events, DB schemas |

## How Specs Flow In

```
Service Repo (on push to main/staging/dev)
  │
  ├─ Build service
  ├─ Export OpenAPI spec
  └─ Push spec to this repo (same branch)
          │
          ▼
This Repo (merge-specs.yml workflow)
  │
  ├─ Lint individual specs
  ├─ Filter out internal routes
  ├─ Merge into unified spec
  └─ Commit merged/openapi.yaml
          │
          ▼
GitHub Pages (deploy-docs.yml workflow)
  │
  └─ Publish merged/openapi.yaml
          │
          ▼
API Gateway (runtime fetch)
  │
  └─ Serves Scalar UI at /docs
     fetching spec from GitHub Pages
```

### Service Spec Sources

| Service | Format | CI Trigger |
|---------|--------|------------|
| authentication-service | JSON (build-time export via `Microsoft.Extensions.ApiDescription.Server`) | `dotnet build` |
| user-account-service | JSON (build-time export) | `dotnet build` |

## Local Development

### Prerequisites

- Node.js 18+ (for `redocly`)
- `npm install -g @redocly/cli`

### Merge Specs Locally

```bash
./scripts/merge-specs.sh
```

### Validate Specs

```bash
./scripts/validate-spec.sh
```

### Preview Docs

```bash
npx @scalar/cli serve merged/openapi.yaml
```

## Adding a New Service

1. Set up spec export in the service's CI (see existing services for patterns)
2. Add the service's spec push step targeting `specs/<service-name>.yaml` or `.json`
3. Update `scripts/merge-specs.sh` to include the new spec in the merge
4. Add a tag for the service in the merge config
5. Push — the merge workflow will pick it up automatically

---

## What's Excluded

The unified spec only covers **public-facing endpoints**. The following are intentionally excluded:

- `/internal/*` — service-to-service routes
- `/health`, `/ready` — infrastructure probes
- `/metrics` — Prometheus scrape endpoints
- Database schemas, RabbitMQ events, deployment configs

These live in each service's own `docs/` directory and `contract.md`.
