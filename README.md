# LegalDispatch API Documentation

> Unified API documentation for the LegalDispatch platform — all public-facing endpoints in one place.

## Overview

This repo holds the **unified OpenAPI specification** and **narrative guides** for the LegalDispatch API. Individual service specs flow in via CI/CD and are merged into a single spec that powers the interactive docs at `api.legaldispatch.dev/docs`.

### Two-Layer Documentation Model

| Layer                                       | Audience                              | Contents                                           |
| ------------------------------------------- | ------------------------------------- | -------------------------------------------------- |
| **Unified Gateway Docs** (this repo)        | Frontend devs, QA, external consumers | Public endpoints only (`/api/v1/*`, `/admin/v1/*`) |
| **Individual Service Docs** (service repos) | Backend devs, DevOps                  | All routes, internal APIs, events, DB schemas      |

## How Specs Flow In

```
Service Repo (on push to main/staging/dev)
  │
  ├─ Build service
  ├─ Export OpenAPI spec
  └─ Push spec to this repo (same branch)
          │
          ▼
This Repo (merge-specs.yml workflow)
  │
  ├─ Lint individual specs
  ├─ Filter out internal routes
  ├─ Merge into unified spec
  └─ Commit merged/openapi.yaml
          │
          ▼
GitHub Pages (deploy-docs.yml workflow)
  │
  └─ Publish merged/openapi.yaml
          │
          ▼
API Gateway (runtime fetch)
  │
  └─ Serves Scalar UI at /docs
     fetching spec from GitHub Pages
```

## Adding a New Service

### 1. Export a Public-Only Spec

**.NET services:** Add a `"public"` OpenAPI document with a `ShouldInclude` filter
that excludes `/internal/*`, `/health`, `/ready`, and `/metrics` paths. Add an
operation transformer to prefix all `operationId` values (e.g. `svc_`). Add
`Microsoft.Extensions.ApiDescription.Server` for build-time export + a
`spec-export` Makefile target.

**Go services:** Create `api/openapi-public.yaml` containing only public paths.
Prefix all `operationId` values with a short service prefix (e.g. `doc_`).

### 2. Create `push-spec.yml` Workflow

Copy an existing service's `.github/workflows/push-spec.yml` and update:

- The spec file name in `specs/` (e.g. `specs/my-service.json`)
- The build command (`.NET: dotnet build`, Go: `redocly bundle`)
- The path trigger filter

The workflow pushes the spec to the matching branch in this repo using the
`DOCS_REPO_TOKEN` PAT.

### 3. Update the Merge Pipeline

In `scripts/merge-specs.sh`:

- Add the new spec to the `SPECS` array
- Add tag mappings to `TAG_MAP` (map service tags → unified tag name)
- Add the unified tag to the `TAGS` array with a description

### 4. Push and Verify

Push to `dev` first. The merge workflow will:

1. Lint the new spec
2. Merge all specs into `merged/openapi.yaml`
3. Validate the merged output
4. Commit and deploy to GitHub Pages

Check the Scalar UI at `/docs` to verify the new endpoints render correctly.

## What's Excluded

The unified spec only covers **public-facing endpoints**. The following are intentionally excluded:

- `/internal/*` — service-to-service routes
- `/health`, `/ready` — infrastructure probes
- `/metrics` — Prometheus scrape endpoints
- Database schemas, RabbitMQ events, deployment configs

These live in each service's own `docs/` directory and `contract.md`.
